20.1a Understanding what is meant by a programming paradigm

  • DefinitionA programming paradigm is a general style of program design that shapes how problems are modeled and solved.
  • OverviewThe four paradigms here are low level, imperative (including procedural), object oriented, and declarative.
  • TipDifferent paradigms provide different abstractions that suit different problems and levels of hardware proximity.
    • DefinitionAbstraction means hiding detail so you can focus on the main idea.
    • AnalogyHardware proximity means working very close to the machine, such as reading or writing specific memory locations.

20.1b Low level programming and addressing modes

  • OverviewLow level programming works close to the processor, using explicit control of memory and registers.
  • DefinitionA register is a fast storage location inside the processor used during calculations.
    • AnalogyThink of a register as a tiny notepad the CPU uses briefly.
    • ExampleDuring a calculation like 23 + 7, the CPU places numbers in registers, adds them, and stores the result back. The registers are the scratch space used for the steps.
  • DefinitionAddressing modes describe how an instruction finds the data it will use.
    • DefinitionImmediate: the data value is part of the instruction itself.
      • ExampleA command includes the number 5 inside it, so no memory lookup is needed.
    • DefinitionDirect: the instruction contains a memory address where the data is stored.
      • ExampleThe command points to address 2000. The CPU goes to 2000 and reads the value stored there.
    • DefinitionIndirect: the instruction points to a memory location that holds another address where the data is stored.
      • ExampleThe command points to address 2000. Location 2000 contains the number 3500, so the CPU then reads the data from address 3500.
    • DefinitionIndexed: the effective address equals a base location plus the value in an index register.
      • ExampleA block starts at base 1000. The index register holds 3. The effective address is 1000 + 3 = 1003, which selects the fourth item in the block.
    • DefinitionRelative: the location is found by moving a few steps forward or backward from the current instruction.
      • ExampleThe current instruction is at address 500. A relative step of +4 moves to address 504 in the program.
  • DefinitionKey terms in plain words:
    • DefinitionEffective address: the final place in memory where the data is found.
    • DefinitionBase location: the starting point for a block of related data.
    • DefinitionIndex register: the number that selects an item within that block.
    • AnalogyApartment address: the building is the base, the flat number is the index, and the full postal address is the effective address.
  • OverviewTypical control flow: a comparison checks a condition; a conditional branch moves to a named step (a label) when the condition is met; a loop repeats steps until a stated condition is met.
    • ExampleRepeat a display action five times: set a counter to 1, display the item, increase the counter to 2, check if it is greater than 5, if not jump back to the display step, and continue until the counter becomes 6, then stop.
  • TipPlan the sequence of memory accesses and register use; trace values step by step.

20.1c Imperative including procedural programming

  • DefinitionImperative programming lists steps to achieve a result, and each step updates the program state in a controlled way.
  • DefinitionState is all the information stored at a moment in time; a side effect is any change to that state that remains after a step finishes.
  • OverviewCore building blocks: sequence, selection, and iteration acting on variables and data structures.
  • TipProcedural programming organises code into named procedures and functions with clear inputs and outputs, which helps reuse and testing.
  • ExampleCalculate the average of seven daily temperatures: write down the seven numbers, add them one by one to a running total, then divide the total by 7 to get the average. Each step updates the state by changing the running total.

20.1d Object Oriented Programming (OOP)

  • OverviewOOP models a program as a set of objects that represent things in the problem. Objects keep their own data and offer operations so other objects can work with them.
  • DefinitionAn object bundles state with the operations that act on that state.
    • ExampleBank account object: state includes the account owner and current balance; operations include show balance, deposit, and withdraw. The operations are the only allowed way to change or read the state.
  • DefinitionA class is a blueprint that defines the attributes and methods shared by its objects. An object is an instance of a class.
    • ExampleBankAccount class: attributes balance and owner; methods deposit, withdraw, and show balance. Instances: Account A for Sara with balance 120; Account B for Omar with balance 50; Account C for Amal with balance 0. All are built from the same blueprint and follow the same rules.
  • DefinitionEncapsulation: keep attributes hidden behind a public interface so state changes happen only through defined operations; this gives safer updates and fewer side effects.
  • DefinitionAccessors read attributes and mutators update them, while applying validation to keep the object valid.
    • ExampleStock item: quantity is kept inside the object. An accessor shows the current quantity. A mutator updates the quantity, but if a negative number is given the mutator refuses it to avoid an invalid state.
  • OverviewObject lifecycle: construction sets a valid initial state and allocates resources; destruction releases resources when objects are no longer needed.
    • ExampleDocument object: on creation, it reserves space and sets the title. When finished, it releases the space so the system can use it for something else.
  • TipDesigning classes: find nouns in the problem text as candidate classes and verbs as responsibilities.
  • TipKeep the class interface minimal and purposeful. Expose only what other parts of the program need to use.
    • DefinitionInterface means the list of operations the class offers.
    • AnalogyA vending machine has many inner parts, but you only use the visible buttons. Those buttons are the machineâ€™s interface.
  • DefinitionInheritance models an is a relationship where a subclass reuses and extends a superclass.
  • DefinitionContainment or aggregation models a has a relationship where one class owns or organises other objects as part of its structure.
  • TipUse inheritance only when the object truly is a base type and can be used in its place; use containment when an object has another as a part.
    • ExampleDog and Animal: a Dog is an Animal and shares Animal features. Car and Engine: a Car has an Engine as one of its parts.
  • DefinitionOverloading allows multiple operations to share a name if their required inputs differ.
  • DefinitionPolymorphism lets different subclasses respond to the same operation name with behaviour that suits their type.
  • ExampleGraphics editor:
    • Polymorphism: the editor stores a list of shapes. It tells each shape to draw. Each shape draws itself in its own way, but the editor uses the same draw instruction for all shapes.
    • Overloading: the editor prints information using one operation name. If the input is a number it prints a number; if the input is text it prints text. The name is the same but the expected inputs differ.
  • TipWrite down always true rules for each object (invariants), what must be true before an operation (preconditions), and what will be true after it (postconditions), so the object stays valid.
    • ExampleBank account: Invariant balance is at least 0; Pre withdrawal requires balance at least equal to the amount; Post withdrawal makes the balance smaller by the amount.
    • ExampleOnline order: Invariant item count is at least 0; Pre packing requires stock at least equal to the requested quantity; Post packing reduces stock by the requested quantity.

20.1e Declarative programming

  • DefinitionDeclarative programming states what must be true rather than how to compute it.
    • DefinitionA knowledge base is a structured list of facts and rules.
  • OverviewA knowledge base contains facts and rules that model the problem domain.
  • OverviewTo solve a task, state a goal or query. The system searches and checks which facts and rules satisfy the goal.
  • TipPipeline: record facts, state rules that combine facts, ask a goal; the system searches for matches and returns answers that satisfy the goal.
  • TipUse precise facts, consistent rules, and clear goals that name domain relationships.
  • ExampleAnimals and suitability:
    • Facts: caracal is wild; carthorse is wild.
    • Rule: an animal is not suitable as a pet if it is wild.
    • Derivation: from the facts and the rule, the system concludes that caracal is not suitable as a pet and carthorse is not suitable as a pet.
  • ExampleFamily tree inference:
    • Facts: Alice is parent of Bob; Bob is parent of Charlie.
    • Rule: a person is an ancestor of another person if they are a parent of that person, or a parent of an ancestor.
    • Derivation: using the facts and the rule, the system marks Alice as an ancestor of Charlie without listing any step by step instructions.
