1. What “input size n” means

  • n means “how much data the algorithm receives.”
  • Examples:
    • If a list has 8 numbers, then n = 8.
    • If a file has 2000 records, then n = 2000.

2. What Big O means (and what it does not mean)

  • Big O describes how the number of steps (or the extra memory) grows when n grows.
  • Big O is not the exact time in seconds, because seconds depend on the computer and the programming language.
  • Big O answers questions like:
    • “If n becomes 10 times larger, how much larger does the work become?”

3. Why “worst case” is often used

  • The worst case is the situation where the algorithm must do the most work.
  • It is used because it gives a safe guarantee: “the algorithm will not be worse than this.”
  • Example (searching in a list):
    • Best case: the item is the first one.
    • Worst case: the item is the last one (or not present), so you check almost all items.

4. Time complexity (Big O for running time)

A. O(1) constant time

  • Meaning: the work stays the same even if n grows.
  • Example:
    • Checking if 24 is even (one simple check).
    • Whether the list has 10 items or 1,000,000 items does not change this check.

B. O(n) linear time

  • Meaning: the work grows roughly in proportion to n.
  • Example (linear search):
    • List length n = 5: in the worst case you may check 5 items.
    • List length n = 10: in the worst case you may check 10 items.
    • Doubling n roughly doubles the checks.

C. O(log2 n) logarithmic time

  • Meaning: the work grows slowly because each step removes a large part of the remaining data.
  • Key idea: each step halves what is left.
  • Example (binary search on a sorted list):
    • n = 16 items
    • Step 1: 16 remaining
    • Step 2: 8 remaining
    • Step 3: 4 remaining
    • Step 4: 2 remaining
    • Step 5: 1 remaining
    • It took about 5 steps, and 5 is log2(16).
  • Practical meaning: even if n becomes very large, the number of steps increases slowly.

D. O(n²) quadratic time

  • Meaning: the work grows roughly like n times n.
  • Typical reason: two levels of repetition (often described as “nested loops”).
  • Concrete example (pair checking):
    • If you compare every item with every other item:
      • n = 4 gives about 4×4 = 16 comparisons
      • n = 8 gives about 8×8 = 64 comparisons
    • Doubling n makes the work about 4 times larger.
  • Example algorithms (worst case): bubble sort, insertion sort.

E. O(n³), O(n⁴), …

  • Meaning: the work grows even faster than n².
  • Concrete example:
    • If a task repeats for every triple of items, growth can be about n×n×n = n³.

F. O(2^n) exponential time

  • Meaning: when n increases by 1, the work can roughly double.
  • Small example (doubling effect):
    • n = 5 -> about 32 units of work
    • n = 6 -> about 64 units of work
    • n = 7 -> about 128 units of work
  • Example: a recursive Fibonacci method can grow exponentially.

5. “Basic operation” (what we count)

  • To estimate time complexity, we often count a repeated “basic step” such as:
    • a comparison (>, <, ==)
    • an assignment
    • a swap
  • Example (bubble sort):
    • A key repeated step is “compare two neighbouring items.”
    • The worst case leads to about 1 + 2 + 3 + … + (n − 1) comparisons.
    • This grows like n², so worst case bubble sort is O(n²).

6. Space complexity (Big O for memory)

  • Space complexity measures how extra memory grows with n.
  • “Extra memory” means memory used beyond storing the input itself.

A. O(1) constant space

  • Meaning: the algorithm uses the same extra memory no matter how large n is.
  • Example:
    • Using only a few variables to compute a result.

B. O(n) linear space

  • Meaning: extra memory grows in proportion to n.
  • Example:
    • Creating an extra array/list that stores n values.

7. How to compare two algorithms that do the same task

  • Compare both:
    • Time complexity (running time growth)
    • Space complexity (extra memory growth)
  • Example (searching):
    • Linear search: O(n) time (worst case checks many items).
    • Binary search: O(log2 n) time (each step halves the remaining items).
  • Conclusion idea:
    • For large n, O(log n) grows much more slowly than O(n), so it scales better.
