1. Why compare algorithms?

  • The same problem can often be solved using different algorithms.
  • Some algorithms are faster than others for large data sets.
  • Some algorithms use more memory than others.
  • We compare algorithms to choose the most efficient solution.

Cambridge requires comparison based on: time taken and memory used.

2. What does input size n mean?

  • n represents the size of the input data.
  • If a list contains 50 items → n = 50.
  • If a binary tree has 200 nodes → n = 200.
  • If sorting 10 000 values → n = 10 000.

Complexity describes how performance changes as n becomes very large.

3. Big O notation

  • Big O notation describes how running time or memory grows as input size increases.
  • It measures the order of growth, not exact seconds.
  • It is normally calculated using the worst-case scenario.

Worst case is used because it guarantees the algorithm will never perform worse than this.

Example (linear search worst case): If the item is last in the list or not present, the algorithm checks all n items → O(n).

Big O ignores constants and small terms. For example: 3n + 10 → O(n)

4. Time complexity (running time)

O(1) — Constant time

  • Running time does not change as n increases.
  • Example: accessing an array element by index.

O(n) — Linear time

  • Running time increases in direct proportion to n.
  • Example: linear search.
  • If n doubles, the work roughly doubles.

O(log n) — Logarithmic time

  • Each step reduces the remaining data significantly (often halves it).
  • Example: binary search.
  • For n = 1 000 000 → about 20 steps only.

O(n²) — Quadratic time

  • Often caused by nested loops.
  • Example: bubble sort (worst case), insertion sort (worst case).
  • If n doubles, work becomes about 4 times larger.

O(2ⁿ) — Exponential time

  • Work doubles when n increases by 1.
  • Example: naive recursive Fibonacci.

5. Example: Bubble sort worst case

In bubble sort, the main repeated operation is: comparing two adjacent items.

Number of comparisons:

  • n = 3 → 3 comparisons
  • n = 4 → 6 comparisons
  • n = 5 → 10 comparisons

Total comparisons = 1 + 2 + 3 + ... + (n − 1)

This equals n(n − 1)/2.

For very large n, we ignore smaller terms → growth behaves like n².

Therefore, bubble sort worst case is O(n²).

6. Space complexity (memory use)

  • Space complexity measures how extra memory grows as n increases.
  • It does not count the input data itself.

O(1) — Constant space

  • Uses only a fixed number of variables.
  • Example: bubble sort, insertion sort.

O(n) — Linear space

  • Uses extra memory proportional to n.
  • Example: creating an additional array of size n.

7. Comparing two algorithms

Example: Searching a sorted list

  • Linear search → O(n)
  • Binary search → O(log n)

If n = 1 000 000:

  • O(n) → about 1 000 000 comparisons
  • O(log n) → about 20 comparisons

For large n, algorithms with smaller growth rates are significantly more efficient.
