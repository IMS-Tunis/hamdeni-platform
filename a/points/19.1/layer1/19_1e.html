1. What “input size n” means

  • n means “how much data the algorithm receives.”
  • Examples:
    • If a list has 8 numbers, then n = 8.
    • If a file has 2000 records, then n = 2000.
    • If searching a binary tree with 100 nodes, then n = 100.

2. What Big O means (and what it does not mean)

  • Big O describes how the number of steps (or the extra memory) grows when n grows.
  • Big O is not the exact time in seconds, because seconds depend on the computer and programming language.
  • Big O ignores constant values and focuses only on the growth rate.
    • Example: 3n + 10 grows like n, so it is written as O(n).
  • Big O answers questions like:
    • “If n becomes 10 times larger, how much larger does the work become?”

3. Why “worst case” is often used

  • The worst case is the situation where the algorithm must do the most work.
  • It is used because it gives a safe guarantee: “the algorithm will not be worse than this.”
  • Example (searching in a list):
    • Best case: the item is the first one.
    • Worst case: the item is the last one (or not present), so you check almost all items.

4. Time complexity (Big O for running time)

A. O(1) constant time

  • Meaning: the work stays the same even if n grows.
  • Example:
    • Checking if 24 is even (one simple check).
    • Accessing an array element using its index.

B. O(n) linear time

  • Meaning: the work grows roughly in proportion to n.
  • Example (linear search):
    • n = 5 → up to 5 checks.
    • n = 10 → up to 10 checks.
    • Doubling n roughly doubles the checks.

C. O(log n) logarithmic time

  • Meaning: the work grows slowly because each step removes a large part of the remaining data.
  • Key idea: each step halves what is left.
  • Example (binary search on a sorted list):
    • n = 16 items
    • 16 → 8 → 4 → 2 → 1
    • About 4–5 steps.
  • The base of the logarithm does not matter in Big O notation.

D. O(n²) quadratic time

  • Meaning: the work grows roughly like n × n.
  • Typical reason: two nested loops.
  • Concrete example (pair checking):
    • n = 4 → about 16 comparisons.
    • n = 8 → about 64 comparisons.
  • Example algorithms (worst case): bubble sort, insertion sort.

E. O(n³), O(n⁴), …

  • Meaning: the work grows even faster than n².
  • Example: three nested loops may give about n × n × n = n³ operations.

F. O(2ⁿ) exponential time

  • Meaning: when n increases by 1, the work can roughly double.
  • Example:
    • n = 5 → about 32 units of work
    • n = 6 → about 64 units of work
    • n = 7 → about 128 units of work
  • Example algorithm: naive recursive Fibonacci.

5. “Basic operation” (what we count)

  • To estimate time complexity, we count a repeated basic step, such as:
    • a comparison (>, <, ==)
    • an assignment
    • a swap
  • Example (bubble sort):
    • The key repeated step is “compare two neighbouring items.”
    • The worst case leads to about 1 + 2 + 3 + … + (n − 1) comparisons.
    • This equals n(n−1)/2, which grows like n².

6. Space complexity (Big O for memory)

  • Space complexity measures how extra memory grows with n.
  • Extra memory means memory used beyond storing the input itself.

A. O(1) constant space

  • The algorithm uses the same extra memory regardless of n.
  • Example: using only a few variables.

B. O(n) linear space

  • Extra memory grows in proportion to n.
  • Example: creating an extra array of size n.

7. How to compare two algorithms that do the same task

  • Compare both:
    • Time complexity
    • Space complexity
  • Example (searching):
    • Linear search: O(n) time.
    • Binary search: O(log n) time.

Concrete scaling example:

  • If n = 1,000,000:
    • O(n) → about 1,000,000 checks.
    • O(log n) → about 20 checks.
  • For large n, O(log n) grows much more slowly than O(n).
