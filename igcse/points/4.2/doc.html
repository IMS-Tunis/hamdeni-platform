4.2a: High-level and low-level languages

  • A high-level language is a type of programming language that uses words similar to human language, such as "if", "print", or "input". These languages are designed to be easy to understand, write, and debug.
    • High-level languages do not require knowledge of the computer‚Äôs hardware or instruction set.
    • They are often portable, meaning they can run on different types of computers.
    • Portable programs run on different computers without modification, because the high-level language is not tied to a specific machine architecture.
    • Programs written in high-level languages are generally easier to maintain over time.
    • Once you learn one high-level language, many of the same programming techniques apply to others like Python, Java, Visual Basic, or Pascal.
  • A low-level language is closer to the computer's own language and is harder for humans to read and write.
    • Low-level languages include machine code (binary) and assembly language (mnemonics).
    • These languages are specific to a computer's hardware and cannot easily be used on other systems.
    • They allow direct manipulation of the hardware, which can lead to faster and smaller programs.
    • Reasons to use a low-level language include faster execution, smaller memory use, and the ability to directly control hardware or use special machine instructions.
    • However, low-level programs take more time to develop and are more difficult to debug.
  • Example: High-Level vs Low-Level Code
    The high-level instruction `Sum := FirstNumber + SecondNumber` is simple and clear. In assembly language, the same task might require:
    ```
    LDA First
    ADD Second
    STO Sum
    ```
    In machine code, this would become a set of binary or hexadecimal instructions like `0001 00010010`. This shows how high-level code is much easier to read and write.
  • Think of high-level languages like giving instructions to a friend: ‚ÄúMake me a sandwich.‚Äù Low-level languages are like saying, ‚ÄúTake two slices of bread, open the peanut butter jar, use a knife...‚Äù : more precise, but definitely more work!

4.2b: Assembly language and assembler

  • Assembly language is a type of low-level programming language that uses short commands (called mnemonics) to represent instructions.
    • Examples of mnemonics include LDA (load), ADD (add), and STO (store), each of which performs a specific operation on memory or data.
    • To understand and use assembly language, the programmer needs to know what each mnemonic means and how the hardware works.
    • Understanding assembly often requires knowledge of hardware parts like the accumulator, which is a special register used to temporarily store values during calculations.
    • Assembly language allows more control over memory use and performance, which can be useful for fast or memory-efficient programs.
    • In most cases, one assembly instruction is directly mapped to one machine code instruction, giving the programmer precise control over hardware behavior.
    • Assembly language must be translated into machine code before it can be executed.
    • In assembly language, the programmer must define and manage the memory addresses of variables manually, which offers precision but increases complexity.
  • A program called an assembler is used to convert assembly language instructions into machine code.
  • Example: Adding Numbers in Assembly
    The following three assembly commands perform the same task as `Sum := FirstNumber + SecondNumber` in a high-level language:
    ```
    LDA First
    ADD Second
    STO Sum
    ```
    This shows how a single high-level command might require multiple assembly instructions.
  • If assembly language were a dance routine, mnemonics would be the choreographer‚Äôs shorthand: ‚ÄúLDA‚Äù means ‚ÄúStep left and pose!‚Äù It‚Äôs more readable than binary, but you still need to know the moves!
  • Example: High-Level Code in Python : `print("Hello!")` is easy to write and understand. It uses an English-like word, making programming feel more natural for beginners.

4.2c: Compilers and interpreters

  • Computers only understand machine code, so programs written by humans must be translated before they can run.
    • A translator is a utility program that converts human-readable code into machine code.
    • Translators are categorized as compilers, interpreters, or assemblers, depending on the type of source code and how the translation occurs.
  • A compiler translates the entire program written in a high-level language into machine code all at once.
    • If there are syntax errors, the compiler stops and produces an error report.
    • A compiler will only create an executable if there are no syntax errors in the program.
    • The error report from a compiler usually includes the line numbers and the type of each error.
    • If there are no errors, it creates an executable file that can be run repeatedly without translating again. This file is ready for distribution and doesn't need the compiler or source code to run.
    • Compiled programs do not need the source code or compiler to run, and the code is protected from being edited.
  • An interpreter translates and runs one instruction at a time from a high-level language.
    • If there is a syntax error, it stops immediately and shows an error message.
    • Interpreted programs need to be translated every time they are run, which makes them slower.
    • Interpreters are often used during development because they allow real-time testing and debugging.
  • An assembler is used to translate programs written in assembly language into machine code.
  • Example: Interpreter vs Compiler Error Reporting : When using an interpreter, the program stops at the first error, allowing immediate fixes. A compiler checks the whole program and shows a list of all errors at once. This means interpreters are more useful during debugging, while compilers are more efficient for finished programs.
  • Example: Compiler Output : A high-level command like `Sum := First + Second` may be translated to:
    ```
    0001 00010010 ; load
    0100 00010011 ; add
    ```
    This shows how multiple binary instructions are produced from a single high-level line.
  • A compiler is like baking a cake and handing someone the finished product. An interpreter is like showing up with the ingredients and making the cake in front of them : every time they ask for a slice!
  • Example: One Instruction, Two Translators
    If a program contains `Sum := FirstNumber + SecondNumber`, a compiler will convert the entire code into binary once. An interpreter will instead read and run this line each time the program is used, making it slower but useful during testing.

4.2d: Advantages and disadvantages of compiler vs interpreter

  • Interpreters are more helpful during program development.
    • They allow programs to be debugged and tested more quickly by running code line by line.
    • The programmer can fix problems as they appear and continue testing from the same point.
    • Interpreted programs are easier to edit and test before they are finished.
    • Interpreters allow testing of incomplete programs, which helps developers find and fix errors early, even before the whole program is finished.
    • An interpreter is useful for debugging because it allows the programmer to test small parts of the code quickly and fix problems as they appear.
    • However, they require the interpreter software each time the program runs, and they are slower because they are retranslated every time.
  • Compilers are more useful when the program is finished and ready for sharing or selling.
    • The compiler translates the entire program into machine code and creates an executable file.
    • The executable runs faster and does not need the compiler or source code to run.
    • A executable program can run without needing the compiler or the original source code, making it suitable for distribution.
    • The executable file protects the source code from copying or editing.
    • However, it takes more time to write, debug, and compile the program during development.
  • Example: Using Translator Tools for a Game
    Saria uses an interpreter while writing a game to test and fix each part quickly. When she finishes, she compiles it into an executable file to send to users. The compiled version is faster, works without extra software, and hides the source code.
  • Using a compiler during development is like wrapping a present without checking what‚Äôs inside : risky! An interpreter lets you peek inside after every step, so surprises are less likely.

4.2e: Role and functions of an IDE

  • An Integrated Development Environment (IDE) is software that helps programmers write, test, and run programs.
    • Some IDEs support one programming language (like PyCharm for Python), while others support many (like Visual Studio).
    • A code editor in the IDE allows the programmer to type and edit code directly.
    • The IDE includes a translator (compiler or interpreter) so the program can be run and tested.
    • The built-in translator in an IDE works together with tools like error diagnostics, version control, and code navigation to help programming.
  • Most IDEs also include debugging tools to help find and fix errors.
    • A runtime environment shows the program‚Äôs output while it runs.
    • A debugger can:
      - run the program line by line (single stepping)
    • IDEs allow step-by-step (line-by-line) execution, which helps identify the exact location where logic errors occur during runtime.
      - stop the program at chosen points (breakpoints)
      - display current variable values (watch window)
  • Many modern IDEs include version control tools, which allow programmers to track changes to their code, undo mistakes, and collaborate with others more easily.
  • IDEs also help while writing code by detecting and correcting errors early.
    • **Error diagnostics** shows mistakes as you type and may suggest fixes.
    • **auto-completion** suggests keywords or variables while typing.
    • **auto-correction** fixes common typing or syntax mistakes.
    • **Auto-documentation** explains the purpose of different parts of code.
    • IDEs may include extra features like bracket matching, version control systems, or auto-documentation tools to manage code changes and formatting.
    • **prettyprinting** formats code and uses colors to make it easier to read.
  • Example: Multi-language IDE : Visual Studio supports C++, C#, and Python, while PyCharm focuses only on Python. This shows how some IDEs are designed for flexibility, while others are optimized for one language.
  • Example: IDE Auto-Correction : If you mistype `prin`, the IDE suggests `print()` and may fix the typo automatically. This helps reduce errors and saves time.
  • Example: Features of a Modern IDE
    In Visual Studio, the code editor shows suggestions as you type (auto-completion), underlines mistakes in red (error diagnostics), and offers fixes (auto-correction). The debugger lets you step through the program and see what values are stored in each variable.
  • IDEs are like smart kitchens for coders : they suggest recipes (auto-completion), stop you if you mix up sugar and salt (error diagnostics), and even help you clean up after (auto-formatting)!
