Section 1

Defining a Single-Table Database

1.1 What is a database?

A database is a structured collection of data.

  • It allows users to store, organise, search and retrieve information.
  • It may store information about people, items or events.
  • A single-table database contains only one table.

1.2 Tables, fields and records

A database table is organised into rows and columns.

FIELD

One specific piece of information

  • A field is represented by a column.
  • Each field should have a meaningful name.
  • Field names are normally written as one word without spaces.

Examples: FirstName, DateOfAdmission, WardNumber

RECORD

All the data about one person, item or event

  • A record is represented by a row.
  • Every record contains the same fields.
  • Records may be added, changed or deleted.

1.3 Defining a database from storage requirements

The objective is to define the structure of a single-table database from the information that needs to be stored.

Read the storage requirements and identify the separate pieces of information that must be stored. Each piece of information becomes a field.

  1. Identify what one record represents.
  2. Give the table a meaningful name.
  3. Identify the required fields and give each one a meaningful field name.
  4. Choose a suitable data type for each field.
  5. Add suitable validation rules.
  6. Identify a suitable primary key.

Converting storage requirements into field names

A wool shop needs to store information about each item. The wording in the requirements is converted into short field names:

The name of the item→ItemName
The colour of the item→Colour
The weight of the item→Weight
The price of the item→Price
The quantity in stock→Quantity
Each record in this table represents one item sold by the shop.

1.4 Validation

Validation checks whether entered data follows the required rules. It helps prevent unsuitable data from being stored.

  • Some checks are performed automatically by database software.
  • Other checks must be set by the database developer.

Section 3

Understanding and Identifying Primary Keys

3.1 Purpose of a primary key

A primary key is a field that uniquely identifies each record.

  • The same primary-key value must not appear in two different records.
  • A primary key allows one exact record to be identified reliably.
  • A person's name is usually unsuitable because more than one person may have the same name.

3.2 Choosing a suitable primary key

A suitable primary key must contain a value that is guaranteed to be different in every record.

  • Ask whether two records could contain the same value. If they could, the field is unsuitable.
  • If no existing field is guaranteed to be unique, add a new identification field.
  • A primary key does not have to be an integer. For example, HospitalNumber = HN123456 may be stored as text/alphanumeric.

3.3 Why BookID is suitable

The visible values in another column may currently look different, but that does not make the field a suitable primary key. Its values must be guaranteed not to repeat in any record.

Author: one author can write more than one book.

Section 4

Reading, Understanding and Completing SQL Scripts

4.1 What is SQL?

  • SQL stands for Structured Query Language.
  • SQL is used to obtain information from a database.
  • A list of SQL commands that performs a task is called an SQL script.
  • A query is a request for specific information from a database.

A complete SQL statement begins with SELECT, identifies a table using FROM, and ends with a semicolon.

4.2 SELECT and FROM

SELECT chooses the fields to display, while FROM identifies the table. Both are required in a basic query, and a semicolon marks the end of the statement.

SQL query

SELECT BookName, Author
FROM Books;

Display the book name and author from every record in the Books table.

4.3 WHERE

WHERE includes only records that meet a condition.

SELECT chooses the columns to display, while WHERE chooses the rows to include.

SQL query

SELECT FirstName, LastName
FROM Orders
WHERE Posted = Yes;

Display the first and last names only for orders that have been posted.

The field used in WHERE does not have to appear after SELECT. In this example, Posted controls which records are included but is not displayed.

4.4 Values and comparison operators in WHERE

Use familiar comparison operators inside a WHERE condition:

=<><<=>>=
  • Write text and character values inside quotation marks: WHERE Consultant = 'Mr Smith'
  • Write integer and real values without quotation marks: WHERE TotalCost < 5.00
  • For Boolean values, follow the form shown in the table: TRUE/FALSE or Yes/No

A text value may appear in a displayed table without quotation marks. In an SQL condition, quotation marks show that the value is text or a character.

4.5 AND and OR in SQL

ORDER BY sorts the query result. Use ASC for ascending order and DESC for descending order. Ascending order is the default.

4.10 Identifying the output of an SQL statement

  1. Identify the table
    Read the name after FROM.
  2. Apply the condition
    Keep the records that satisfy WHERE.
  3. Sort if required
    Apply ORDER BY.
  4. Display or calculate
    Use the fields, SUM or COUNT written after SELECT.
