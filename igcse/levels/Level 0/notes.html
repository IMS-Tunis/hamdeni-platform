1. Understand each phase of the program development life cycle

The program development life cycle is the standard process used to build software and it is composed of four steps analysis, Design, Coding then Testing

  1. Analysis identifies the problem and produces a clear requirements specification (a short, written list of what the program must do and how it will be checked).
    • Abstraction means keeping only essential details and removing non-essential information.
    • Decomposition splits a large problem into smaller parts (sub-tasks or modules, where a module is a small part of a program that does one job, for example “calculate total”).
    Example: Abstraction in everyday life

    Think of maps for different purposes. A road map shows roads and junctions; a rail map shows lines and stations. Each map removes details that are not needed for its task so the essential information stands out.

    Example: Decomposition of a routine

    Getting dressed can be broken into smaller, ordered steps:
    1) choose items
    2) remove current clothing if needed
    3) put on items in a sensible order.
    In code, these could be functions such as chooseItems() and putOnShirt().

  2. Design plans the solution using the approved specification from analysis.
    • Decomposition continues to organise tasks and their interactions.
    • The design states what each part should do, how it should do it, and how parts fit together.
    • The design is recorded with structure diagrams (tree diagrams that show parts under the whole), flowcharts (diagrams of steps using standard shapes), and pseudocode (code-like English). These 3 tools will be the focus of all next levels.
  3. Coding implements each module in a suitable programming language (like python, java or VB.NET) after the algorithms are designed.
    • Iterative modular testing is carried out during development.
    • Code is amended until each module works as required.
  4. Testing checks the completed program using varied test data to confirm correctness and that requirements are met.
    • Use normal (typical), abnormal (invalid), extreme (at the ends), and boundary test data (just inside/just outside the allowed limits).
    • Testing confirms that all tasks work together as designed.
    Example: Age input data sets

    Allowed whole numbers 10 to 100
    Normal: 10, 50, 100
    Abnormal: 9, −1, 101, "age"
    Extreme: 10 and 100
    Boundary pairs: 9 vs 10; 100 vs 101

    Example: Password length data sets

    Minimum 8 characters
    Normal: "abcdefgh"
    Abnormal: "1234567"
    Extreme: exactly 8 characters
    Boundary pair: 7 vs 8 characters

2. Understand that every computer system is made up of sub-systems, which are made up of further sub-systems

  • A computer system can be divided into sub-systems and then into smaller sub-systems until each unit performs a single, clear action.
    • A computer system typically involves hardware, software, data, communications, and people (users and technicians who operate or maintain the system).
    • Top-down design means starting from the whole app, then splitting it into screens or features, and repeatedly splitting each feature until a single action remains, like “save file”.
    • Structure diagrams show this hierarchy as a tree; each box can later become a subprogram, function, or procedure.

3. Understand problem decomposition into Inputs, Processes, Outputs, Storage

  • Problems can be decomposed into Inputs, Processes, Outputs, and Storage (IPOS).
    • Inputs: data entered while the system runs.
    • Processes: computations or decisions on current inputs and previously stored data.
    • Outputs: information shown or printed for users.
    • Storage: data kept for use during and after execution (variables, constants, arrays); persistent storage saves data in files on a device such as a hard drive, SSD, or USB stick.
    Example: Alarm app (IPOS)

    Input: set or clear alarm time; press snooze
    Process: check current time; manage snooze timing
    Output: play alarm sound
    Storage: store the alarm time(s)

4. Understand the purpose of algorithms

  • An algorithm is the ordered list of steps to complete a task; it makes the required processes explicit.
    • Common processes include linear search, bubble sort, totalling, counting, minimum/maximum, and average calculations.
    Example: Alarm-time algorithm

    Imagine you are coding a phone alarm that rings at 07:00.
    A simple algorithm can:
    1) wait a short interval (for example, 10 seconds)
    2) read the current time
    3) compare it with the stored alarm time
    4) play the alarm sound when they match

    Example: Average of a set of marks

    To compute the class average:
    1. decide the list of marks to include;
    2. set Total to 0 (initially);
    3. add each mark to Total one by one;
    4. at the end divide Total by the number of marks;
    5. report the resulting value as the average.

    Example: Minimum value in a list

    To find the smallest temperature in a list:
    1. start by assuming the first temperature is the minimum;
    2. compare each remaining temperature with the current minimum;
    3. whenever a smaller value is found, replace the current minimum with it;
    4. after the scan finishes, report the final minimum.

  • Algorithms are commonly shown as flowcharts or pseudocode. A trace table supports a dry run to infer purpose and detect logic errors using sample data.

5. Understand and use different methods to design solutions

  • Flowcharts show the steps of an algorithm in order using standard symbols.
    • Key symbols: Terminator (rounded rectangle for Start/End), Process (rectangle), Input/Output (parallelogram), Decision (diamond), and directed flow lines (arrows).
  • Pseudocode is a language-independent, code-like description of an algorithm that uses meaningful names.
    • It is not limited by strict programming syntax but is precise enough to convert into code.
    • Core constructs include:
      Assignment (←), Selection (IF..THEN..ELSE..ENDIF; CASE OF..OTHERWISE..ENDCASE), Iteration (FOR..TO..NEXT; REPEAT..UNTIL; WHILE..DO..ENDWHILE), and Input/Output (INPUT, OUTPUT).
  • Trace tables help follow variable values step by step during a dry run to infer purpose and detect logic errors using sample data.
